Project Requirements

Detailed analysis and documentation of the Portalt project requirements

Partner Introduction and Project Background

Our project is a collaboration with IBM and NHS to develop a white-labeled VR/AR platform designed to revolutionize touring experiences. The project emerged from the need to create more accessible and engaging virtual tours for various industries, particularly for innovation center spaces.

IBM provided technical expertise and infrastructure support, while NHS offered insights into healthcare applications and accessibility requirements. UCL Computer Science department served as the academic partner, providing research guidance and educational use case perspectives.

The background of this project stems from the increasing demand for virtual experiences accelerated by global events that limited physical access to spaces. Organizations across various sectors expressed the need for customizable, interactive virtual tour solutions that could be adapted to their specific requirements.

Project Goals

  • Create a white-labeled VR/AR visiting & activity system for innovation center spaces
  • Develop an intuitive interface that allows non-technical users to create and customize virtual tours
  • Build a two-sided system with an admin dashboard for customization and an end VR/AR experience
  • Support multiple devices and platforms for maximum reach
  • Provide interactive elements that enhance engagement and information retention
  • Create a scalable solution that can grow with user needs and technological advancements

Requirement Gathering

How we collected requirements

Stakeholder Interviews

We conducted in-depth interviews with representatives from IBM, NHS, and potential end-users to understand their needs and expectations.

Market Research

We analyzed existing VR/AR tour solutions to identify gaps and opportunities for innovation.

User Surveys

We designed and distributed surveys to potential users across different sectors to gather quantitative data on preferences and requirements.

Heuristic Evaluation

We chose to carry out heuristics evaluation as it is quick to complete in our limited time frame.

Survey Design and Analysis

We designed a comprehensive survey using a mixed-methods approach, combining Likert scale questions, multiple-choice options, and open-ended questions. The survey was distributed to potential users across healthcare, education, real estate, and tourism sectors.

Survey Analysis Methods:
  • Quantitative analysis using statistical methods
  • Thematic analysis for qualitative responses
  • Cross-tabulation to identify sector-specific needs
  • Priority mapping to identify critical features
Key Insights:
  • Users valued customization options
  • Many prioritized ease of use over advanced features
  • Cross-platform compatibility was highly desired
  • Healthcare users emphasized accessibility features
  • Education sector prioritized interactive elements

Personas

Based on our research, we identified the following typical users of our platform:

Admin User

Tech-Savvy Professional

Goals: Create and customize virtual tours for their organization or clients. Manage content and user access.

Challenges: Needs an intuitive interface to efficiently create professional-looking tours without extensive technical knowledge.

Preferences: Values customization options, ease of use, and the ability to update content regularly.

End User

VR/AR Experience Visitor

Goals: Experience virtual tours with engaging, interactive content. Easily navigate and access information.

Challenges: May have varying levels of technical expertise and comfort with VR/AR technology.

Preferences: Prioritizes intuitive navigation, clear information presentation, and immersive experiences.

MoSCoW Requirements

Functional Requirements

Priority ID Requirement
Must Have F1 The system must allow administrators to create virtual tours by selecting environments.
F2 The system must allow administrators to add and position 3D objects within the virtual environment.
F3 The system must provide an interface for end users to navigate through virtual tours.
F4 The system must allow end users to view information about objects in the tour.
F5 The system must support both VR and AR viewing modes.
Should Have F6 The system should allow administrators to customize branding elements of the tour.
F7 The system should provide analytics on user engagement with tours.
F8 The system should allow administrators to import custom 3D models.
F9 The system should provide templates for common tour scenarios.
Could Have F10 The system could allow for multi-user experiences within the same virtual tour.
F11 The system could provide voice narration capabilities for tours.
F12 The system could include gamification elements like quizzes or challenges.
Won't Have F13 The system won't have built-in 3D modeling capabilities.
F14 The system won't have real-time collaborative editing features in the initial release.

Non-Functional Requirements

Priority ID Category Requirement
Must Have NF1 Usability The system must have an intuitive interface that requires minimal training for both administrators and end users.
NF2 Performance The system must maintain a minimum of 60 FPS in VR mode to prevent motion sickness.
NF3 Compatibility The system must be compatible with major VR headsets (Oculus, HTC Vive) and AR-capable mobile devices.
NF4 Accessibility The system must include basic accessibility features like text-to-speech and high-contrast options.
NF5 Security The system must secure user data and provide authentication for administrative access.
Should Have NF6 Scalability The system should support at least 100 concurrent users per tour without performance degradation.
NF7 Reliability The system should have an uptime of at least 99.5%.
NF8 Maintainability The system should be built with a modular architecture to facilitate updates and extensions.
Could Have NF9 Localization The system could support multiple languages through localization.
NF10 Extensibility The system could provide an API for third-party integrations.
